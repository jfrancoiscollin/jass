# Jass GitOps runner

A dedicated machine (typically a Hetzner CCX cloud server or AX
dedicated server) becomes a self-driving Jass runner: it polls the
`jobs/queue/` directory in this repo every 5 minutes and runs the
scripts you commit there. Output, status and small artefacts are
committed back to `jobs/results/<id>/`.

You never need to SSH back in unless something breaks.

## One-time setup on a fresh Ubuntu 24.04

```bash
ssh root@<server-ip>
curl -sSL https://raw.githubusercontent.com/jfrancoiscollin/jass/main/infra/bootstrap.sh | bash
# Final step it prints: open the GitHub deploy-key URL and paste the
# public key with "Allow write access" checked.
exit
```

That's it. The runner systemd timer is now ticking every 5 min.

## Scheduling a job

Each job is a shell script in `jobs/queue/`. Pick a numeric prefix so
ordering is obvious. Commit + push. The runner will pick it up within
5 minutes.

```bash
# Example: generate a small depth-20 dataset
cat > jobs/queue/0001-smoke.sh <<'EOF'
#!/usr/bin/env bash
# id: 0001-smoke
# description: hello-world job, validates the runner end-to-end
set -euo pipefail
cd /root/jass
echo "host=$(hostname)"
echo "nproc=$(nproc)"
echo "cpu=$(lscpu | grep 'Model name' | head -1)"
echo "jass-version="
echo "position fen W:W31-50:B1-15" | ./build/jass | head -3
EOF
chmod +x jobs/queue/0001-smoke.sh
git add jobs/queue/0001-smoke.sh
git commit -m "queue smoke job 0001"
git push
```

## Reading results

```
jobs/results/<id>/
├── status.json         { state, exit_code, started_at, ended_at, host }
├── output.log          tail of stdout+stderr (max 1 MB)
└── artefacts/          small files (< 50 MB each) the job produced
```

Files > 50 MB stay on the server (path noted in `status.json` under
`artefacts_server_path`). For bigger artefacts, upload to object
storage from inside the job script.

## Debugging

```bash
# On the server (only if something is wrong):
systemctl status jass-runner.timer    # timer health
systemctl status jass-runner.service  # last tick result
journalctl -fu jass-runner.service    # live logs of the tick
cat /root/jass/jobs/state/in-flight.json  # currently running job (if any)
tail -f /root/jass/jobs/results/<id>/output.log.raw  # raw job output
```

## Pausing the runner (GitOps, no SSH)

To stop the runner from picking new work — for example before adding
a second machine, to avoid races on a single-host runner — commit a
`jobs/state/runner-paused` flag and push:

```bash
mkdir -p jobs/state
echo "paused on $(date -u)" > jobs/state/runner-paused
git add jobs/state/runner-paused
git commit -m "pause runner"
git push
```

Within 5 min the runner picks up the change and starts no-op'ing
ticks. In-flight work, if any, keeps running and is still reaped /
heartbeated normally — pause only affects new pickups.

To resume:

```bash
git rm jobs/state/runner-paused
git commit -m "resume runner"
git push
```

This replaces the previous "schedule a `systemctl disable` via a
detached `at`/`systemd-run`" hack: the flag is fully version-
controlled and reversible by anyone with write access to the repo,
no SSH required.

## Killing a stuck in-flight job (GitOps, no SSH)

If a job hangs (e.g. an engine wedged on a bad command), commit an
empty `jobs/state/kill-in-flight` flag and push:

```bash
mkdir -p jobs/state
touch jobs/state/kill-in-flight
git add jobs/state/kill-in-flight
git commit -m "kill stuck job"
git push
```

On the next tick the runner SIGTERMs the wrapper's whole process
group, waits 2 s, SIGKILLs survivors, then deletes the flag (so a
future job isn't killed by a leftover flag). The tick after that
reaps the dead job and marks it `failed` in `status.json` with the
normal exit-code path. To re-queue with a fix: edit the script, then
`rm -r jobs/results/<id>/` and push.

## Conventions

- One job runs at a time (the runner reserves itself via
  `jobs/state/in-flight.json`). Long jobs span many ticks; subsequent
  ticks are no-ops until the in-flight process exits.
- A queue entry is "done" iff `jobs/results/<id>/status.json` exists
  with `state` in `{completed, failed}`. Re-queue a failed job by
  deleting its result directory.
- The runner pushes via the dedicated `~/.ssh/jass_runner` deploy
  key generated by `bootstrap.sh`. Don't put a PAT in the repo.

## Multi-host setup (`JASS_HOST_FILTER`)

By default every runner picks every job in `jobs/queue/`. Running two
or more hosts against the **same** repo without coordination causes
races on `in-flight.json` and heartbeat commits — symptoms include
"one host marks the other's running job as failed" and "every other
heartbeat commit fails the push because the remote moved".

The `JASS_HOST_FILTER` env var solves this with a filename-prefix
filter. Each host only picks scripts whose stem starts with its
filter value. Two hosts with non-overlapping filters never race.

One-time setup per host (SSH required once):

```bash
ssh root@<ccx23-ip>

# Pick a unique prefix matching the jobs intended for THIS host.
# Convention: 0020a- for host A, 0020b- for host B, etc.
mkdir -p /etc/jass-runner
cat > /etc/jass-runner/host.env <<'EOF'
JASS_HOST_FILTER=0020a-
EOF
chmod 0600 /etc/jass-runner/host.env

# Wire it into the systemd unit:
systemctl edit jass-runner.service
# In the editor, paste:
#   [Service]
#   EnvironmentFile=/etc/jass-runner/host.env
# Save + exit.

systemctl restart jass-runner.timer
exit
```

The runner logs the filter on every tick so you can confirm it's
picked up: `runner: JASS_HOST_FILTER='0020a-' keeps 1/12 candidate jobs`.

When `JASS_HOST_FILTER` is unset, behaviour reverts to "pick every
job" — i.e. the single-host default. So this is fully back-compatible.

### Putting it to use: the 2× CCX23 pilot

`jobs/templates/0020a-multihost-pilot-host-a.sh` and
`jobs/templates/0020b-multihost-pilot-host-b.sh` are dormant siblings
that each generate a 100K-record half of a 200K dataset. To run the
pilot:

1. Spin up two CCX23 hosts, bootstrap both via `infra/bootstrap.sh`.
2. SSH into host A → set `JASS_HOST_FILTER=0020a-` (procedure above).
3. SSH into host B → set `JASS_HOST_FILTER=0020b-`.
4. `git mv` both 0020a/0020b templates from `jobs/templates/` to
   `jobs/queue/`. Commit + push.
5. Each host picks only its own job at the next tick. They run in
   parallel (~12-15 h each).
6. When both finalize: `scp` host-b.bin to host A, then `git mv`
   `jobs/templates/0021-merge-multihost-shards.sh` to
   `jobs/queue/`. Host A (or whichever host has both files) picks
   the merge job and produces the combined 200K JNNW dataset.

That validates the multi-host architecture end-to-end. Same pattern
scales to N hosts for the future 10M run (each gets a distinct
filter; the merge step concatenates N files).
