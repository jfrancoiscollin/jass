# Jass NNUE training pipeline

End-to-end procedure for training a `LinearNetwork` (the evaluation
plug-in shipped in [`src/nnue.cpp`](../src/nnue.cpp)) from data
generated by the engine itself.

The pipeline has three steps, all standalone:

1. **Generate** a binary dataset of (position, target-score) records
   by self-play.
2. **Train** a linear model on that dataset (Python + NumPy).
3. **Load** the resulting weights into a `LinearNetwork` from C++.

A single dependency on the Python side: NumPy.  PyTorch is optional —
useful only if you extend the trainer to a non-linear model.

---

## Step 1 — Generate training data

```sh
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j

./build/jass --gen-data 100000 selfplay.bin
```

This plays self-play games (from random openings, depth 4 for the
move advance, depth 8 to label sampled positions) and writes a
binary file containing roughly 100 000 (position, score) records.

Approximate cost on this machine: ~0.5 ms per record on a single
thread, so a 1 M-record dataset takes around 8 minutes.  Run a
shorter generation first to validate the format and the round-trip.

### Wire format

```
Header (8 bytes)
    4 bytes  magic = "JNNT"
    4 bytes  uint32  record_count

Each record (37 bytes)
    8 bytes  uint64  white_men   bitboard
    8 bytes  uint64  white_kings bitboard
    8 bytes  uint64  black_men   bitboard
    8 bytes  uint64  black_kings bitboard
    1 byte   uint8   side_to_move   (0 = white, 1 = black)
    4 bytes  int32   target_score   (centipawns, side-to-move POV)
```

All multi-byte fields are little-endian.

---

## Step 2 — Train

```sh
python3 tools/train.py --data selfplay.bin --out nnue.bin
```

Outputs `nnue.bin`: 800 bytes containing 200 little-endian int32
weights, one per (square, piece-kind) cell, in the layout
`LinearNetwork::load()` expects.

The trainer is a closed-form least-squares fit (NumPy's `lstsq`).
For 1 M records and 200 features it converges in under a second.
To experiment with non-linear models, replace `fit_linear` in
[`train.py`](train.py) with a PyTorch `nn.Module` and a standard
training loop.  The weight-export step at the end stays the same as
long as the network architecture matches `LinearNetwork`.

### What the trainer reports

Sample output:

```
loading selfplay.bin …
  100000 records, 200 features
fitting linear model …
weights summary: min=-128 max=145 mean=+0.4 std=68.4
  mean(white-man  ) = +110.7
  mean(white-king ) = +297.2
  mean(black-man  ) = -111.0
  mean(black-king ) = -298.5
wrote weights to nnue.bin (800 bytes)
```

The trained means should hover around `±MAN_VALUE` for men and
`±KING_VALUE` for kings (i.e. ±100 / ±300 with the current
constants in [`src/eval.hpp`](../src/eval.hpp)).  Big deviations
suggest the dataset is too small or skewed.

## Step 3 — Load the weights into the engine

From C++:

```cpp
#include "nnue.hpp"

jass::LinearNetwork net;
if (!net.load("nnue.bin")) {
    std::cerr << "failed to load nnue.bin\n";
    return 1;
}
const int score = net.evaluate(pos);   // STM-POV centipawns
```

The default-constructed network is the *handcrafted-equivalent*
linear model.  Loading a trained file replaces those weights wholesale.

To make the search itself use the trained network (rather than the
handcrafted `evaluate()`), swap the call in
[`src/search.cpp`](../src/search.cpp) — see
[`docs/EXTENDING.md`](../docs/EXTENDING.md) recipe #4 for the
mechanical change.

## Smoke test

After running the pipeline once, you can verify the round-trip on a
known position:

```cpp
#include "nnue.hpp"
#include "position.hpp"

const auto p = *jass::Position::from_fen("W:W31-50:B1-20");

jass::LinearNetwork stock;     // default-constructed, handcrafted-eq.
jass::LinearNetwork trained;
trained.load("nnue.bin");

std::cout << "stock="   << stock.evaluate(p)
          << " trained=" << trained.evaluate(p) << "\n";
```

In a typical run the two are within a few tens of centipawns of
each other on the start position.  If `trained.evaluate(p)` is wildly
off (hundreds of centipawns), check that the dataset was generated
with `use_book(false)` — book moves return score 0 and skew the
average.

## Reproducibility

`--gen-data` uses a fixed RNG seed (`0x5eed5eed5eed5eedULL`), so the
same engine binary will produce the same dataset for the same
record count.  This is helpful for unit-testing the format and for
A/B comparisons of evaluation-side changes.  To get truly varied
data across runs, edit the seed in [`src/main.cpp`](../src/main.cpp)
or pass a different command-line seed (open issue: not yet wired up
to a flag).
